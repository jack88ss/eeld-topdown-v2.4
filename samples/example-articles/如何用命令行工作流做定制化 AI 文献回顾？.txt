## 工作流

其实这个工作流本身并不复杂，它只包含了四个步骤而已。让我们一起来探索这个精简但强大的工作流程。

我在这里把它定义成了一个 YAML 文件。YAML 是一种人类友好的数据序列化标准，它的语法简洁明了，非常适合用来描述配置信息。

可见，这里包含了一个工具调用，和 3 个大语言模型处理步骤。这种组合充分利用了工具的特定功能和大语言模型的灵活性，让我们能够处理复杂的任务。接下来，我们将详细解析每个步骤的作用和实现方式。

## 大纲生成



这是大纲生成部分的大语言模型配置说明。



![[assets/BA55ED3EDEBA67C75A4718CC4084C98D.png]]



这里我们使用的模型是 Anthropic 提供的 Claude 3.5。这个模型在推理和指令遵从方面是目前最强的模型之一，另外从性价比上讲，也要比 Claude 3 Opus 好。



这里面我们定义的输入是前一步 EXA 文献检索的结果。然后我们执行 academic outliner 这样的对应的 pattern 系统指令。我们把它生成的结果，定义为 outline，也就是即将生成的大纲。后续步骤可以直接援引其内容，进行进一步的处理加工。





下面我们就打开 academic outliner 看它具体的内容。



![[assets/1FCE608B562140896903B85423E61F81.png]]



我们现在介绍的是一种提示词技巧，称为结构化提示词（structured prompt）。它包含了多个精心设计的部分。



![[assets/57AE26A47DD4E85E388204F9A872C20B.png]]





第一部分被称为「身份和目标」（Identity and Purpose）。你初次看到这个提示词时，可能会感到有些惊讶。

老师，为什么一开始就说模型是一个有 20 年丰富工作经验的学术期刊资深编辑呢？明明它只是一个大语言模型啊。

这是因为在这个部分，我们首先要给大语言模型一个明确的角色定位，告诉它在这个任务中应该扮演什么样的角色。大语言模型在训练过程中接触到了海量的、各种类型的资料。对于模型来说，不同身份的人（比如学者、记者、作家）撰写的内容权重相当，它可以在回答中应用任何一种视角或者风格。

但是，当我们需要模型**执行特定类型的任务时**，这种区分就变得非常重要了。考虑到我们现在要执行的是一个学术性质的任务，如果不给大语言模型一个明确的学术角色定位，它输出的内容风格可能来自于剧本、人物传记、新闻，甚至是儿童故事……总之会与我们的预期有很大差距。它可能会用过于口语化的语言，或者逻辑上缺乏学术写作所需的严谨性。

因此，在提示词的开始，**我们一定要给大语言模型定义一个明确的角色**。通过这种方式，我们实际上是在为模型设定一个框架。从此以后，它会以这个特定角色的视角、口吻和经验来筛选材料，并生成符合这个角色设定的内容，提高输出内容的相关性和质量。

定义了角色之后，我还进一步说明了这个角色的具体**技能**：它擅长从各种杂乱的资料中理出头绪。这是在进一步引导模型，明确告诉它我对其输出的期望是什么。

接下来我要说明**目标**：我会给你输入一些资料，你需要一步步思考，帮助我妥善形成简体中文的标题与大纲。你看，在写作提示词过程中，一定要保持清晰详细。因为在这种情况下，大语言模型更有可能记住并且准确遵从你的指令。



![[assets/6E8AC81045E7176B23A0FD6505FAFF23.png]]



下一步，我们在结构化提示词中给它提供输出指令（Output Instructions）。这里我们有两个主要要求：首先，输出主题；其次，输出大纲。你会发现，确定主题相对简单，因为返回的搜索结果中已经包含了相关问题。我们只需要指示模型从这些结果中提取出即可。

我们要求模型使用 Markdown 的一级标题格式来输出主题。这样能清晰地突出主题，还能为整个文档建立一个基本的层级结构。使用 Markdown 格式的好处是，它既易于阅读，又便于后续的编辑和转换。

确定主题后，下一步是让模型输出大纲。我们刚刚使用了exa工具搜集信息，它除了为我们获取检索到内容文本的前1000个字符外，还返回了多个高亮内容。这些信息虽然丰富但较为杂乱，要将它们系统化，我们需要充分发挥大语言模型的能力。我们要让它调动阅读、理解、组织和规划等多方面技能，帮助我们构建一个**结构清晰、层次分明的大纲**。

尽管我们只检索了 10 篇文献，但由于内容十分丰富，我们决定在二级标题下增设三级标题。这样的结构设计能为我们提供更细致的分类指引，使得每一条资料都能找到其对应的合适位置。

关于三级标题的设置，我有一个重要原则：要么不设置；如果要设置，则每个二级标题下至少要包含两个三级标题。因为这样才能把结构有效支撑起来。



![[assets/C19C5000FC2741F14830B51C2604E22C-1.png]]

接下来，我对大语言模型强调，避免产生任何不实的内容。具体来说，它提供的所有标题都必须源自对搜索结果的仔细阅读和精准提炼，绝不允许擅自进行无根据的总结或猜测。

大语言模型有时会根据用户的意图自行生成内容，这在文献回顾时尤其令人担忧。它可能会编造各种内容和参考文献，严重影响研究的可信度和准确性。为了避免这种情况，我明确要求模型不要产生任何虚构内容，而必须严格依据我们提供的资料来生成内容。这样做可以确保我们的文献综述建立在真实、可靠的研究基础之上。



![[assets/12B66838D1FD90C1C9CF843CB92D3B47.png]]



对输出格式，我们明确要求使用Markdown格式，并指示模型在**生成大纲后立即停止输出**。这个要求非常重要，因为我们计划将这一步的结果用作下一步操作的输入或部分输入。如果不加限制，模型可能会在输出大纲前生成额外的内容，比如说「好的，根据你的指令，我已经为你生成了完整的大纲，下面请看。」或完成大纲输出后，添加类似这样的话：「以上就是我为你输出的大纲。如果你有进一步的需求，欢迎提出，我会继续帮助你修改和完善。」这些情况，相信你之前已经见识过了吧？

这种额外的信息对我们的文献回顾工作当然是多余的，甚至可能会严重干扰后续流程的自动化处理。因此，通过明确的停止指令，我们可以确保输出的纯粹，以便为下一步的操作做好准备。



![[assets/785FC5A71BE85C13483816FD3547305D.png]]

接下来，我们告诉模型，要把后面的内容当作输入（Input）来处理。

这就是我们结构化提示词包含的各项内容。在今后，如果你为了严肃的任务撰写提示词时，也不妨参考其架构和相应技巧。





## 内容填充

下面我们来看内容填充环节。与前一环节相似或者重复（例如 Input）的部分，咱们就不再赘述。



![[assets/7F69C0BA191A91B6C80678B7999642EC.png]]



显然，这一步骤中，大语言模型的角色设定发生了变化。它变成了「一位资深科研助理，有 20 年的丰富工作经验。你擅长把各种杂乱的资料理出头绪，并且不浪费任何一点宝贵的信息来源」。

我们这样设定是因为大语言模型后续要将这些资料分门别类地放入我们之前看到的大纲中。如果它处理得不够细致，很多资料可能会被忽略。这对我们来说是一种损失。

任务的要求是：下面我给你一些搜索来的资料，这些是由搜索引擎直接生成的结果（也是我们要填充的内容），以及一个主题和大纲（这是我们的架构，我们准备按照它的层级结构来进行内容的填充）。

下面这一句我提到「你一步步思考」（Think step by step），这是在启动它的「思维链」(Chain of Thoughts, CoT)，提升大模型思考的效能。然后我提出的要求是「帮我把资料按照要求填充到大纲里，形成草稿」，这是我们对它的**目标设定**。



![[assets/D63C3ACA0D988E6D6DBA714531663136.png]]



在输出指令中，我们详细规定了大语言模型的处理步骤：

首先，要求模型照抄大纲内容。这样可以保持文档的层次结构，便于后续的内容扩展。

其次，把与刚刚列出的标题相关的所有搜索资料全部原文照录于此处，不需要你进行任何的改动、润色甚至是翻译，保留对应的来源 URL。这一点非常重要 —— 我们必须要保持它的原文链接，这样的话我们在后续才能方便地进行验证。

这种详细的指示反映了人与大语言模型交互时的特点。我们需要提供清晰、具体的指令，以确保模型能够准确理解并且照章执行我们的需求。不过，实话实说，有时会显得较为啰嗦。😂

![[assets/C31DBA95D06C8827F699C2650E8F689D.png]]

下一步是输出格式。我们要求模型继续按照 Markdown 格式组织输出。我们依然要求模型避免添加多余的开头和结尾。这样可以确保输出的内容简洁明了，直接聚焦于我们需要的信息。

另外与大纲输出相比，我们这里多了一条要求——要大语言模型避免产生无关内容或进行不必要的总结。因为在中间处理过程中，我们最担心的是信息的大幅损失。大语言模型最容易造成信息损失的情况嘛，往往就是它**自作聪明尝试总结**时。





## 回顾撰写

下面是最为关键和令人兴奋的阶段 —— 我们要让大语言模型生成最终的文献回顾稿件了。



![[assets/166F35E11676D2B6206ABADA4BB25F4D.png]]



这次咱们给模型设定的角色是一位资深教授，特别擅长利用科研助手整理的资料。这样可以有效承接上一步的输出内容。

之后，我们明确定义了模型的目标：利用整理后的资料，结合原始的搜索结果，撰写一个完整的文献综述章节。接下来，我们会看到具体的输出指令。



![[assets/3973B0BC5309DF0913A0DB9EEE54AF1F.png]]



这个指令要求我们以输入内容为资料来源，使用符合学术界读者阅读习惯的语言进行写作。「简洁明快」是我们期望看到的文献回顾的语言风格。这里我们要求它撰写一个 2000 字左右的文献回顾小节。

需要注意的是，这并不意味着大语言模型真的恰好输出2000字。我们提出2000字的建议，还是为了避**免模型过度总结内容**。大语言模型通常倾向于提供简洁的总结，这可能导致内容过于精简，造成大量信息的损失。而设定较高的写作字数，则可以让大语言模型「松弛」一些，尽量避免强行总结压缩的冲动。

在文献回顾中，我们要求写作内容有观点、有论证，还要提供证据。这里我要求逻辑严谨，层次分明。如果遇到专业术语，可以保留原文，但在第一次提及时需要提供清晰的解释。这样做可以确保文献回顾的专业性和可读性兼具。对于非专业读者来说，这种做法尤其有帮助。

最后，我要求它不允许使用感叹号。在文献回顾写作中，这看似是多余的说明，因为对你来说它是尝试。但面对大语言模型，我们还是希望避免它使用过于随意或情感化的表达方式，以保持文献回顾的**专业性和客观性**。



![[assets/5D2D99C75C7C0176BFF4C0630E9EEADA.png]]



你会发现，在这个阶段，「输出格式」这部分很长。



我们要输出的，是文献综述的一个章节。我们希望形成一个完整的小节，而不是将内容分割成多个过短的小节。所以此处我们指明，要求只包含一个 Markdown 二级标题，和若干中文自然段落。因为如果内容不够，却强行分成一堆小节，则每一部分的篇幅过短，支撑不起一个小节应该有的篇幅。

我规定如果提出论点，必须提供充分的论证。回顾一下，我们之前的样例输出中，每个观点后面都附有详细的论证过程，就是靠这个要求来实现的。



![[assets/F5D198465B1794E84C5B74437A2588B8.png]]

特别强调的是，如果有具体的数据，一定要将其纳入论据中。这就是为什么你会在文献回顾中看到诸如 “35%” 等具体数字。

另外如果你不做明确要求，大语言模型很可能会把引用链接统一放在文末，而不对应文稿的具体位置。这会给咱们后续的检验造成麻烦。因此我们明确要求：不要浪费原文中提供的链接，要将它们作为引用放入对应的内容中。这样，你就能在文献综述中看到论述后紧接着就是相应的引用。

我们再次强调如果没有特定内容的来源信息，不要编造。这一要求是为了避免生成虚假或不准确的信息，保证文献综述的真实性和可靠性。

「如果引用的原文不是中文，请翻译。」这是对于非中文文献内容的处理形式。

我们还指出「使用自然段落进行表述，不要采用列表形式」。不这么写的话，大语言模型总爱在那儿「一、二、三、四……」。看似清晰，但这不符合文献回顾的期望体例。

关于来源信息的标注，我们要求在引用位置采用文内 markdown 链接的方式。你会看到文中出现各种蓝色显示的 Markdown 链接，如 ScienceDirect 和 arXiv，点击后可直接跳转到原文。就是这样要求得出来的。

我们再次了强调，不允许使用 '\[1\]'、'\[2\]' 这样的参考文献引用方式，也不要将链接集中放在文末。因为根据模型能力的差异，有些提示词能否遵从，得看你是不是放在了合适的位置，以及进行了足够多次的强调。

跟人说话，咱们还遵从着「重要的事情说三遍」的规律。何况是 AI 呢？对吧？



## 资源获取



前面我们对工作流的四个步骤，分别进行了详细讲解。下面我们来看看具体的执行过程。



首先你需要先了解命令行工作流的运行方式，请你 [参考这篇文章](https://articles.zsxq.com/id_kmwc76rvs578.html)。你需要下载（或者更新）命令行工作流运行文件。



回顾一下，第一种方式是通过 Git 命令克隆仓库：

```

git clone https://github.com/wshuyi/workflows_with_litellm_pub.git

```



如果你已经克隆过仓库了，此时可以进入本地克隆的目录，执行：



```

git pull

```

这种方式我个人比较推荐。因为如果后续有更新，直接拉取更新更加容易。

第二种方式是从 Github 下载项目最新版打包 ZIP 文件，地址是：

<https://github.com/wshuyi/workflows_with_litellm_pub/archive/refs/heads/main.zip>

第三种方式是从 Notion 页面下载：我已经将更新后的压缩文件上传到了 Notion 页面，[请你从这个链接下载](https://wise-pullover-00f.notion.site/workflow-with-lite-0b1890f1ad364f53afe8afc1aa6da54b?pvs=4)。



![[assets/98EC7E8B938EC2D9B86A30338324E3C4.png]]

下载或者更新项目内容后，请你终端下进入该目录，并且执行：



```

poetry install

```

这样可以保证依赖包更新到最新。

![[assets/14F6E227AE51E468BA82578F28162FED.png]]





之后，请你 [到下面这个 Notion 页面，下载本次补充的几个步骤对应的提示词 patterns](https://wise-pullover-00f.notion.site/AI-898ddfee3e654814bae075c79d9b6b28?pvs=4)。没错，这次的几个大语言模型执行步骤的系统提示词命令，我没有公开发布。作为付费用户，你可以下载并且单独添加进去。



![[assets/3BC995EC84CA3A6838ACA2368437EE17.png]]



解压之后，你会发现里面包含了 3 个文件夹。

![[assets/D94E0ABD626688684B563D06B8B6ACD0.png]]

把它们放在命令行工作流目录的 patterns 文件夹下，就可以了。



你需要把文献回顾的问题和年份写在一个文本文件里。例如我把它存放在了 [[`literature-review-query.md`]] 。内容为：



```

query: 大语言模型对各行各业带来的冲击

```



执行：



```

poetry run python app.py "/Users/wsy/Downloads/literature-review-query.md" --workflow=literature_review_exa

```



注意你存储文件的路径，可能和我的不同，所以你需要把上面例子中引号内的路径，改成你自己机器上实际的位置。





提醒一句，别忘了在 .env 文件里，要同时填写 OpenRouter 的 API key，以及 EXA_API_KEY。不填写后者，你就无法调用 exa 工具来搜集信息。切记。



![[assets/DECABF2A3D6EB8E3410896B6050A4EA6.png]]

## 小结



这篇文章我为你介绍了如何用命令行 AI 工作流做定制化的文献回顾。与 Perplexity 中输入相同的提问来对比，它可以返回更为详细的信息，数据更为详实。



每一步的提示词我都已经为你做了详细的讲解，所以你可以在自己的需求基础上加以剪裁。



我觉得有必要再来嘱咐你一句——AI 是我们的帮手，不应该成为你的枪手。因此，在拿到文献回顾的初稿后，你应该尽到一个作者的责任，先要对里面所有引用文献的内容加以核对和查证。之后再根据自己文献回顾的主题，对内容加以调整。**千万不要把 AI 生成的结果，直接端出去给你的导师，甚至是投稿到出版社。**



在你后续使用的过程中，如果有什么经验和心得，欢迎你写在留言区，咱们一起交流讨论。也欢迎你把自己修改后的更符合你自己需求的提示词分享出来，让更多的小伙伴可以从中受益。



祝定制化 AI 文献回顾工作流使用愉快！
