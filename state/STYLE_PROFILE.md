# 风格缓存（更新于 2025-09-17）

> 样例来源：`samples/example-articles/` 目录内的 13 篇个人博客。除非该目录增删文章或作者风格出现明显变化，否则直接复用本节内容。

## 语气特征
- **视角与称呼**：第一人称经验叙述与第二人称对话并行，经常以“我”“你”“我们”快速建立亲密感。
- **节奏**：首两段常用设问或惊讶句带出主题；平均句长约 30 字，但穿插 10–15 字短句强化节奏。
- **情绪基调**：好奇、友好、略带调侃；在工具评测段落转为中立分析，再以鼓励语气收束。

## 结构套路
1. **开场**：生活化情境 + 设问或惊喜。
2. **痛点铺陈**：1–2 段描述读者可能遭遇的困惑，辅以个人例子。
3. **发现 / 方案**：主体 3–4 段拆解流程、体验或对比，常以“首先/然后/最后”串联。
4. **推演 / 反思**：至少 1 段讨论限制、适用人群或对旧方式的取舍。
5. **行动建议 / 邀请互动**：总结收获，并以问句或行动清单收尾。

## 语言习惯
- **设问频率**：每篇约 8–10 个问号，用于段首引导或段尾追问。
- **比喻**：喜用日常物品、学习场景（如“像给未来的自己留张便签”）。
- **关键词**：高频出现在“工作流”“检索”“笔记”“体验”“成本”等语域。
- **强调方式**：偏好行内加粗或引用块；极少使用 emoji，偶尔使用破折号拖长语气。
- **代表句式**：
  - 开场设问：“你有没有这样的时候…？”、“我昨晚又遇到了一个熟悉的坑。”
  - 类比说明：“这感觉就像给未来的自己留下一张醒目的便签。”
  - 行动鼓励：“不妨今晚就试试，把第一条语音笔记录下来。”

## 风格负面清单
- 避免使用带有“规范性/学术论文”语气的句式（如“本文将从三个维度进行论述”）。
- 不堆叠长列表；如需列举，用短句或段内逗号表达。
- 避免过多技术术语缩写，首次出现请附解释；英文专有名词保持原样，但勿拆成单字符（例：保留“GraphRAG”，不要写成 “Gr / ap / h”）。
- 不插入 emoji 大量堆砌；若需要突出情绪，使用文字描述或引用块。

## 图文策略
- 每篇约 10–20 张截图/示意图，使用 `![[assets/...]]` 语法，图后紧跟解说段。
- 图片集中在流程说明或界面演示部分，以“见上图”“如下图所示”串联。
- 保持截图与正文的汉字风格一致，无需额外字体处理。

## 使用指南
- `stylist` 默认勾选“沿用缓存”并在 `state/LOG.md` 记录；仅在样例更新时逐条复核并重写本文件。
- 若用户额外提供风格样例，请先放入 `samples/example-articles/` 再由 stylist 更新此缓存。
- `writer` 建议在草稿中引用上述代表句式或结构作为提示，`editor` 审稿时以“设问频率、类比、行动号召”三项逐条核对。
