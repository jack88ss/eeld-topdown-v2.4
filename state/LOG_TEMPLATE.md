# 日志模板 - 剧本式记录

> 每个代理的工作都应该像剧本一样记录，让读者能够理解整个创作过程。

## 传统格式（保留用于快速记录）
| 时间 (UTC) | Actor | Stage | Change | Evidence |
|------------|-------|-------|--------|----------|

## 剧本式格式（推荐）

```markdown
===== [时间戳 UTC] | [代理名] 登场 =====

【[代理名] 登场】
[自我介绍，说明性格特点和今天的任务]

【观察输入】
- [列出接收到的材料]
- [当前状态和限制条件]
- [特殊要求或风险点]

【内心独白】
[详细的思考过程，包括：
- 对输入的理解和感受
- 面临的挑战和困惑
- 可能的解决方案评估
- 为什么选择特定方案]

【决策过程】
[明确的决策点和理由：
1. 关键决定及其依据
2. 放弃的方案及原因
3. 风险评估和缓解措施]

【执行动作】
✅ [完成的任务1]
✅ [完成的任务2]
⚠️ [遇到的问题或风险]
❌ [失败的尝试]

产出物：
- [文件1]：[简要说明]
- [文件2]：[简要说明]

【问题与移交】
[传递给下一个代理的关键信息]
[需要特别注意的事项]
[未解决的问题]

【[代理名] 退场】
[总结感想，对下一位的寄语]
=====
```

## 多代理协作对话

当需要记录代理间交互时：

```markdown
【Coordinator → Researcher】
"[协调员的请求或指示]"

【Researcher 回应】
"[研究员的反馈或疑问]"
```

## 符号约定

- ✅ 成功完成
- ⚠️ 警告/风险
- ❌ 失败/阻塞
- 🤔 思考中
- 💡 灵感/创意
- 📝 待办事项
- 🔄 正在处理
- ⏰ 时间敏感
- 🎯 关键目标

## 示例记录

```markdown
===== 2025-09-18 10:00:00 UTC | Stylist 登场 =====

【Stylist 登场】
我是风格分析师，像一位细腻的导演，负责把握文章的整体基调。
今天要为一篇关于AI开发工具的技术文章定调，需要保持作者一贯的亲切风格。

【观察输入】
- 主题：规格驱动开发工具
- 样例：13篇个人博客，上次分析是3天前
- 要求：3000字深度文章，需要技术性与可读性平衡
- 风险：技术术语可能让普通读者望而却步

【内心独白】
翻看这些样例，我感受到作者是个喜欢分享的人，总是用"你"来称呼读者，
像朋友间的对话。技术文章容易写得生硬，但作者的风格恰好能中和这种生硬。
那个"像给未来的自己留便签"的比喻用得真好，我要确保这种生活化的表达延续下去。

3000字的要求有点挑战，以往文章多在1500字左右。需要调整节奏，
不能让读者感到冗长...也许可以用更多案例和故事来支撑？

【决策过程】
决定沿用现有风格缓存，因为：
1. 样例未更新，风格保持连贯性很重要
2. 亲切对话风格正好软化技术内容
3. 但需要为长文特别调整节奏控制

关键调整：
- 设问分布从6-10个增加到10-15个
- 段落呼吸感要更强，避免信息轰炸
- 每个技术概念后跟生活化类比

【执行动作】
✅ 复用 state/STYLE_PROFILE.md（v1.2）
✅ 更新 state/WRITING_SPEC.md 增加长文节奏指导
⚠️ 标记风险：需要coordinator帮助简化技术术语
📝 建议：writer多用故事串联，避免纯技术描述

产出物：
- state/STYLE_PROFILE.md（未修改，继续沿用）
- state/WRITING_SPEC.md（v2.0，新增3000字指导）

【问题与移交】
Coordinator请注意：
- 技术术语需要通俗化处理，建立术语表
- 3000字要求需要更多素材支撑
- 建议增加2-3个深度案例

【Stylist 退场】
风格基调已定，像往常一样亲切但更有深度。
Coordinator，接下来看你如何组织素材了。记住，我们要让读者感觉
是在和一个懂行的朋友聊天，而不是在读技术手册。加油！
=====
```

## 注意事项

1. **真实性**：记录真实的思考，包括困惑和失败
2. **可读性**：像写故事一样生动，但不要过度修饰
3. **实用性**：确保信息对问题追踪有帮助
4. **连贯性**：每个代理的记录要承上启下
5. **适度原则**：详细但不冗长，200-500字/代理为宜
