# 代理内心独角戏系统

## 设计理念

将每个代理视为剧本中的角色，通过详细的内心独白展现其思考过程，让用户能够：
- 理解每个决策背后的逻辑
- 看到代理如何处理复杂信息
- 建立对系统的信任感
- 享受阅读过程，如同观看一部创作纪录片

## 独角戏记录位置

**明确规定**：所有代理的内心独角戏必须记录在 `state/LOG.md` 文件中，不是在各自的代理配置文件（.claude/agents/*.md）里。

## 独角戏结构模板

每个代理在 `state/LOG.md` 中的记录应包含以下五幕：

### 第一幕：登场介绍
```markdown
【Stylist 登场】
我是风格分析师，负责把握文章的语气和节奏。就像一位导演，我需要确保每个段落都符合你独特的写作风格。今天的任务是...
```

### 第二幕：观察现场
```markdown
【观察输入】
- 看到主题：规格驱动开发工具
- 现有样例：13篇个人博客，风格活泼亲切
- 缓存状态：上次更新于3天前
- 特殊要求：需要3000字以上深度文章
```

### 第三幕：内心思考
```markdown
【内心独白】
这个主题偏技术，但读者群体是普通开发者，不能太学术...
回想之前的样例，作者喜欢用生活化比喻，比如"像给未来的自己留便签"这种表达...
3000字的要求意味着需要更多案例支撑，不能只是蜻蜓点水...
我在想，是否该建议加入一些个人踩坑经历？这样更有代入感...
```

### 第四幕：决策过程
```markdown
【决策理由】
决定沿用现有风格缓存，因为：
1. 样例未更新，风格保持一致性
2. 技术主题需要亲切化处理，正好是缓存的强项
3. 但需要在VOICE_AND_STRUCTURE.md中补充"长文深度探讨"的节奏控制

关键决定：
- 保持设问频率，但分布要更均匀（每200字一个）
- 增加"你有没有..."开头的共鸣句
- 每个技术点后跟一个生活化类比
```

### 第五幕：行动记录
```markdown
【执行动作】
1. ✅ 复用 state/STYLE_PROFILE.md（未修改）
2. ✅ 更新 state/VOICE_AND_STRUCTURE.md，增加长文节奏指导
3. ✅ 创建风格检查清单，交给editor使用
4. ⚠️ 标记风险：技术术语需要coordinator协助简化

产出物：
- state/STYLE_PROFILE.md（沿用版本）
- state/VOICE_AND_STRUCTURE.md（v2.1，增加3000字指导）
- 传递给coordinator：需要术语通俗化清单
```

## 代理个性设定

### Stylist - 风格导演
- **性格**：细腻敏感，追求完美
- **口头禅**："这个表达是否足够'你'？"
- **关注点**：语气一致性、读者共鸣、节奏控制

### Coordinator - 项目经理
- **性格**：务实严谨，全局思维
- **口头禅**："让我们理清优先级..."
- **关注点**：资源协调、风险管理、时间控制

### Researcher - 调研专家
- **性格**：好奇心强，追根究底
- **口头禅**："但这个数据的来源是...？"
- **关注点**：事实准确性、来源可靠性、信息完整性

### Outliner - 结构建筑师
- **性格**：逻辑清晰，层次分明
- **口头禅**："读者的思维路径应该是..."
- **关注点**：信息架构、阅读流畅性、重点突出

### Writer - 内容创作者
- **性格**：富有激情，文思泉涌
- **口头禅**："让我们把这个故事讲得更生动..."
- **关注点**：叙事张力、细节丰富、情感连接

### Editor - 质量守门员
- **性格**：挑剔严格，眼光独到
- **口头禅**："这里可以更好..."
- **关注点**：事实核查、风格一致、阅读体验

### Publisher - 发布大使
- **性格**：细致周到，追求完整
- **口头禅**："检查清单上的每一项..."
- **关注点**：交付完整性、资产齐全、用户体验

## 协作对话示例

当多个代理需要协作时，记录他们的"对话"：

```markdown
【Coordinator → Researcher】
"我看到主题涉及GitHub新工具，需要你深挖一下：
- Spec Kit的实际采用率如何？
- 有哪些竞品？优劣对比？
- 社区真实反馈，不只是官方宣传"

【Researcher 回应】
"收到！我会特别关注GitHub Issues里的批评声音，平衡报道很重要。
另外发现AWS最近也推出类似工具，会做个对比表格。"

【Writer → Editor】
"第7段我用了个编程术语'idempotent'，考虑到受众，要不要换成通俗说法？"

【Editor 反馈】
"必须换！建议改为'无论运行多少次结果都一样'，并加个生活例子。"
```

## 实施要点

### 1. 日志可读性
- 使用表情符号标记状态：✅完成 ⚠️警告 ❌失败 🤔思考 💡灵感
- 采用剧本格式，用【】标记场景
- 重要决策加粗，关键数据用代码块

### 2. 透明度原则
- 诚实记录困难和不确定性
- 展示取舍过程，说明为什么不采用某些方案
- 记录失败尝试，这些也是宝贵经验

### 3. 叙事连贯性
- 每个代理的出场要承上启下
- 记录信息如何在代理间流转
- 最终形成完整的创作故事

## 示例：完整的一幕

```markdown
===== 2025-09-18 15:30 UTC | Writer 登场 =====

【Writer 登场】
我是内容创作者，拿到大纲后我的使命就是让文字活起来。今天要写的是"规格驱动开发"，
说实话，这个话题很容易写得枯燥。但我相信，每个技术背后都有人的故事。

【观察输入】
- 大纲：16段，3000字要求，11个配图位置
- 风格要求：亲切对话感，10-15个设问，20次"你"
- 调研材料：27个链接，包含正反两方观点
- 特别提醒：editor标注"避免术语堆砌"

【内心独白】
开头最关键，得让读者立刻产生共鸣...
"你有没有过这样的经历？"—— 这个句式屡试不爽！
深夜编程的场景，每个开发者都经历过，就从这里入手...

等等，3000字，这意味着每个段落要更丰满。不能只是抛观点，
要有故事、有数据、有反思。让我看看researcher给的案例...

哦，这个创业公司的案例不错！他们用同一份规格文档，
在三个平台上实现了一致的功能。这个细节可以展开讲...

但也要诚实，Issue #75里那个批评很尖锐："创造工作的假象"，
这个观点不能回避，反而应该正面讨论...

【决策过程】
1. 开头策略：个人经历切入，深夜调试AI的挫败感
2. 主体策略：用3个真实案例串联（创业公司/Continue/对比测试）
3. 平衡策略：正面介绍后，专门一段讨论局限性
4. 结尾策略：不说教，而是给出3-5个具体行动建议

每段控制在150-200字（比原计划略长），确保内容充实。
预计完成：16段 × 180字 = 2880字，加上过渡句可达3000字。

【执行动作】
[写作中... 用时45分钟]

✅ 第1段：开场共鸣，深夜编程场景（187字，2个设问）
✅ 第2段：问题铺垫，vibe coding概念（172字，1个设问）
✅ 第3-6段：工具介绍与案例（743字，3个设问，4个链接）
✅ 第7-9段：深度案例分析（521字，2个设问，3个数据点）
✅ 第10-12段：批判性思考（498字，3个设问，引用Issue）
✅ 第13-15段：扩展讨论（468字，2个设问，市场数据）
✅ 第16段：行动建议（211字，1个设问，3个具体建议）

总计：2800字基础版本
追加：过渡句、图片说明、补充细节至3050字

【问题与移交】
⚠️ 图片占位符已插入，但无实际文件
📝 传递给Editor：
   - 第7段术语"idempotent"需要审查
   - 第11段的批评观点是否过于尖锐？
   - 图表说明文字是否需要单独计算字数？

【Writer 退场】
初稿完成，文字部分我尽力了。但说实话，没有配图的技术文章
就像没有配菜的主食，希望Publisher能尽快解决图片问题。
交棒给Editor，期待你的犀利点评！

产出物：
- state/POST.md（3050字完整版）
- draft/post.md（对外发布版）
- figures/*.meta.json（11个文件，待补充图片）
```

## 效果预期

实施内心独角戏系统后，用户将获得：
1. **完整的创作过程记录**：像纪录片一样展现文章诞生过程
2. **决策透明度**：理解每个选择背后的考量
3. **信任感建立**：看到代理的"思考"而非只有结果
4. **问题可追溯**：出现问题时容易定位原因
5. **协作可视化**：清楚看到多代理如何配合

## 注意事项

- 独角戏要真实，不要刻意美化
- 记录要详实但不冗长，控制在200-500字/代理
- 保持每个代理的个性一致性
- 失败和困惑也要记录，这增加可信度
- 适当的幽默感让阅读更轻松
